What it is

You choose where the answer comes from

Two toggles sit next to the chat box. They decide which retrieval tools the agent is even allowed to use this turn — so the answer is always grounded in a source you picked, never a silent guess.

My Library

📚 My Library

Answers come from your PDF books — embedded with Ollama and searched directly over Pinecone — plus the MongoDB database via MCP. Name a book and it narrows the search to just that title.

Web

🌐 Web

Answers use live web search via the Tavily API. The source URLs are captured and rendered as clickable, persisted "Sources" chips on the answer.

Web Library

Both off

No retrieval tools are wired in — the agent answers from the model's own knowledge. If it doesn't know, it says so and suggests enabling 🌐 Web instead of hallucinating.

What's inside

Built for grounded, honest answers

Every capability exists to keep the assistant accurate and transparent — grounding answers in sources you control, and telling you exactly where each answer came from.

Conversation memory

Turns are persisted in MongoDB via a direct Go driver. A token-budgeted tier keeps recent turns verbatim and folds older ones into a rolling summary, so prompts stay cheap on small local models.

Native PDF search

No external service. The agent embeds your query with Ollama (nomic-embed-text) and queries Pinecone directly for the most relevant book chunks — ported from the Omni-RAG retrieval code.

Book-title narrowing

Mention a book in chat and it resolves against a Mongo books catalog, filtering the vector search to just that title. Ambiguous? The agent clarifies — "do you mean X or Y?" — instead of guessing.

Web citations

Tavily source URLs are captured during the ReAct loop and rendered as clickable, persisted "Sources" chips — so every web-backed claim links back to where it came from.

Honest provenance labels

Each answer carries a tag for exactly where it came from: from My Library, from Web, from Database, or the model's own knowledge. No source pretends to be another.

Drag-and-drop ingestion

A dedicated ingest page takes a PDF, then extracts → chunks → embeds → upserts it into Pinecone and a Mongo books catalog — adding it to My Library for the agent to cite.

Read-only recall tool

A dedicated recall_history tool, scoped to the current conversation and injected by the backend, lets the model fetch exact earlier messages — never the destructive MongoDB CRUD tools.

Dual LLM backend

Runs fully local on Ollama by default; switches to Anthropic Claude when ANTHROPIC_API_KEY, ANTHROPIC_MODEL, and ANTHROPIC_CREDIT_BALANCE: true are all set — no code change.

MongoDB over MCP

A standalone MongoDB Model Context Protocol server exposes CRUD as discoverable tools over SSE. Gated behind My Library, and kept separate from the deterministic conversation-history layer.

System design

How it all fits together

Two decoupled Go services and a pluggable LLM brain, joined by an open protocol. The agent reasons and picks its tools per turn; the MCP server exposes MongoDB as discoverable tools; nothing is hardcoded between them.

The ReAct loop, end to end

A single endpoint runs the full reason → act → observe cycle. The active tool set is assembled per request from the source toggles; tool observations live only inside one turn and are never persisted.

Token-budgeted memory

Recent turns are kept verbatim within a token budget; older turns fold into a rolling summary. Raw tool observations (Pinecone chunks, web text) are never persisted or replayed — re-retrieved fresh each turn if needed.

Per-turn tool gating

The active tool set is rebuilt every request: recall_history is always in; search_pdf + mcp only with My Library; web_search only with Web. The system prompt branches its guidance to match.

Get running

One-command setup

A single script installs and verifies Go, MongoDB, and Ollama, pulls the models, builds both modules, and creates your config files. The only thing you provide is API keys (Tavily / Pinecone / optionally Anthropic).

# macOS · Homebrew · idempotent — safe to re-run
git clone https://github.com/dharmendrra/agentic-ai.git
cd agentic-ai
./setup.sh

# Then start the two services in separate terminals:
cd mcp    && go run .      # MongoDB MCP server on :8083
cd agents && go run .      # Agent + chat UI on :8082

# Open the UI
open http://localhost:8082            # chat
open http://localhost:8082/ingest.html # add PDFs to My Library

What setup.sh automates

Installs/verifies Go, MongoDB (and starts the service), and Ollama; pulls nomic-embed-text + a chat model; runs go mod download and builds both modules; copies config.example.json → config.json in agents/ and mcp/ if missing.

What you provide

API keys can't be auto-provisioned. The script reads TAVILY_API_KEY, PINECONE_API_KEY, PINECONE_HOST (and optionally Anthropic) from your environment if set — otherwise it writes placeholders and prints exactly which keys to fill in agents/config.json. config.json stays gitignored.
